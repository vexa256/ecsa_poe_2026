4.1 FOUNDATION — TAILWIND-ONLY, SHADCN-IDENTICAL

───────────────────────────────────────────────────────────────────
4.2 APPLE — CLARITY, DEFERENCE, DEPTH, CRAFT
───────────────────────────────────────────────────────────────────

1. CLARITY — every element has one purpose; hierarchy via
   weight/size/spacing, not decoration.
2. DEFERENCE — UI steps aside so the CASE / TRAVELLER / DATA
   is the hero. Whisper-soft chrome.
3. DEPTH — layering communicates hierarchy; motion has physics
   (ease-out in, ease-in out).
4. CRAFT AT THE PIXEL — 1px matters; optical balance beats
   mathematical balance; no rough edges.

───────────────────────────────────────────────────────────────────
4.3 MICROSOFT FLUENT — LIGHT, DEPTH, MOTION, MATERIAL, SCALE
───────────────────────────────────────────────────────────────────

1. LIGHT — soft illumination; focus rings; subtle active glows.
2. DEPTH — clear z-axis (base → card → elevated → modal →
   toast); shadows scale with depth.
3. MOTION — 150 ms micro / 250 ms standard / 400 ms max;
   motion explains causality.
4. MATERIAL — surfaces feel real; acrylic translucency only
   on sticky headers and command bars.
5. SCALE — built for density (tables, timelines, logs) without
   ever feeling cramped.

───────────────────────────────────────────────────────────────────
4.4 GOOGLE MATERIAL — STRUCTURE, MEANING, DATA-VIZ, RESPONSIVE
───────────────────────────────────────────────────────────────────

1. MATERIAL AS METAPHOR — surfaces stack, move, respond.
2. BOLD, GRAPHIC, INTENTIONAL — strong type hierarchy,
   confident whitespace.
3. MOTION PROVIDES MEANING — animations explain relationships.
4. DATA VISUALISATION EXCELLENCE — every chart teaches:
   clear axes, restrained palette, annotations, rich
   tooltips. No chart is decoration.
5. RESPONSIVE ADAPTATION — mobile is first-class, never a
   shrunk desktop.

───────────────────────────────────────────────────────────────────
4.5 TESLA — REDUCTION, COMMAND, AMBIENT INTELLIGENCE, REAL-TIME
───────────────────────────────────────────────────────────────────

1. RUTHLESS REDUCTION — if an element does not serve the
   next decision, it does not ship.
2. COMMAND SURFACE — every screen answers: "What can I do
   right now, and what is the system telling me?"
3. AMBIENT INTELLIGENCE — the system anticipates; the right
   action surfaces without the user hunting.
4. MONOCHROMATIC CONFIDENCE WITH ACCENT — predominantly
   neutral canvas; colour used SEMANTICALLY (risk, status,
   progress), NEVER decoratively.
5. REAL-TIME FEEL — counters tick, statuses update,

LAW 7 — NEVER EXPOSE DATABASE ENUMS OR TIER LABELS
   Do NOT say "Suspected Tier 1", "tier1_always_critical",
   "IHR_T1", "VHF_PROTOCOL_ACTIVATED", or any raw DB enum
   to the end user. Resolve enums to their human meaning.
   Resolve suspected cases to the actual DISEASE CATEGORY
   stored in the DB (e.g. "Suspected Viral Haemorrhagic
   Fever — Ebola / Marburg / Lassa"). Use the
   DiseaseIntel.php service as the translation layer.
   Memorise this rule permanently.

LAW 8 — SURFACE 100% OF CASE + TRAVELLER INTELLIGENCE
   Every alert/case view must display the full intelligence
   available in the database about:
     - the suspected case (by disease category, not tier)
     - the traveller (identity, journey, direction,
       document, nationality, contact, destination)
     - every field entered during secondary screening
       (profile, symptoms with onset dates, vitals,
       exposures with YES/NO/UNKNOWN, outbreak context,
       IHR risk, differential ranking with confidence
       bands, matched hallmarks, score breakdowns, WHO
       case definitions, officer overrides, actions taken,
       suspected diseases, disposition, follow-up plan)
     - who is currently working on the case
     - what has happened so far
     - what has been delayed and why
     - collaborators, comments, evidence, handoffs,
       breaches, external responders, case sheet
   Alert views must feel like an intelligent case brief —
   never a log dump, never a vertical data list.

LAW 9 — TAB-AND-WIZARD ORIENTED, NEVER VERTICAL LISTS
   Views are structured around tabs, steppers, and wizards.
   Vertical useless data lists are banned. Aggressively
   code wizards — every link, every action, every decision
   path is wizard-guided with a clear start and finish.

LAW 10 — CLOSED-LOOP VIEW DESIGN
   Every view has:
     - a clear purpose (shown at the top)
     - a clear explanation of what the screen is for
     - a clear set of options
     - a guided decision path
     - a clear finish
     - confirmation of what happened
     - a safe next step
   No view leaves the user wondering what to do.

LAW 11 — EVERY FUNCTIONALITY MUST ACTUALLY WORK
   Every link, every button, every form, every filter, every
   action in every view must be WIRED TO LIVE BACKEND
   FUNCTIONALITY. Tested. End-to-end. No placeholder
   href="#". No dead buttons. No "coming soon." No UI that
   looks polished but isn't wired. Example of the current
   failure we are correcting: /admin/alerts/10 today is
   a view with no working behaviour — UNACCEPTABLE.

LAW 12 — RBAC LIVES IN LARAVEL, NOT IN VIEWS
   RBAC enforcement lives in:
     - route middleware (auth, role, scope)
     - controller authorisation (Gate, Policy, authorize())
     - form requests (authorize() method)
     - policies (api/app/Policies/**)
     - query scoping (global scopes, explicit where clauses
       by poe_code / district_code / pheoc_code /
       country_code pulled from user_assignments where
       is_primary=1 AND is_active=1)
   Views render whatever the controller gives them. Views
   may conditionally show/hide UI CHROME based on
   $user->can('…') or a view-model flag set by the
   controller — but authorisation decisions are NEVER made
   in Blade. If a view needs a capability check, the
   controller computes it and passes it in.

LAW 13 — PUBLIC-HEALTH LANGUAGE, BASIC FOR ANYONE
   Every label, tooltip, helper text, empty state, and
   error message uses plain public-health language. No DB
   field names. No enums. No internal codes. No Laravel
   exception text. No health-surveillance jargon.

LAW 14 — LOADING, EMPTY, ERROR, AND SUCCESS STATES
   Every view designs all four states deliberately:
     - LOADING — skeletons that match the final layout,
       not blank spinners.
     - EMPTY — illustrated, explain why it's empty,
       suggest the next action.
     - ERROR — calm, specific, actionable; never a raw
       stack trace.
     - SUCCESS — confirmation, what happened, what's next.

LAW 15 — ACCESSIBILITY IS NON-NEGOTIABLE
   WCAG 2.1 AA minimum:
     - Colour contrast 4.5:1 body / 3:1 large.
     - Every interactive element is keyboard reachable
       and focus-visible.
     - ARIA roles, labels, and live regions where
       appropriate (aria-live="polite" for guidance,
       "assertive" for critical flags).
     - prefers-reduced-motion honoured.
     - Icons paired with text labels.

LAW 16 — PERFORMANCE BUDGET
   - First meaningful paint < 1.5 s on 4G.
   - Chart libraries lazy-loaded per view that needs them.
   - Livewire polling intervals justified (15 s for
     cockpit, 30 s for hubs, on-demand elsewhere).
   - No N+1 queries (view models pre-compute everything).
